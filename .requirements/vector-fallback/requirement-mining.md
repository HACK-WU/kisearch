# 需求挖掘报告：路径向量语义兜底

> 需求编号：vector-fallback
> 创建时间：2026-06-14
> 状态：需求确认

---

## 原始需求描述

将 Wiki 的路径信息存入向量，当用户传入的 group 路径或 relation 名称没有匹配到时，用向量做近似匹配，返回实际的路径信息，并提示用户这是近似匹配的结果，以提高 Agent 查询体验。

---

## 痛点分析

### 核心问题
当前 ki 命令的路径匹配是**纯字符串精确匹配 + 有限的顶层前缀补全**，无法处理语义层面的近似匹配。

### 根因链
```
路径匹配只支持精确匹配 + 前缀补全
  → AI Agent 凭记忆/推断传入路径，容易有偏差
    → 匹配失败 → 列出候选项 → Agent 再试 → 消耗 token + 轮次
      → 首次命中率低，用户体验差
```

---

## 假设验证（POC 测试）

### 测试环境
- 存储后端：mem CLI (memory-lancedb-pro)
- Embedding 模型：Qwen/Qwen3-Embedding-8B (SiliconFlow API)
- 测试数据：从 `kb/monitor/relations-cache.json` 提取 12 条样本
- 隔离方式：`--tags ki-path` / `--tags ki-relation` + `--scope monitor`

### 测试结果

| 测试场景 | 用户输入 | 正确路径 | Rank 1 | Score |
|---------|---------|---------|--------|-------|
| 精确路径 | `告警系统设计/告警收敛机制` | 同左 | 命中 | ~0.90 |
| 两层漏字 | `告警设计/收敛机制` | `告警系统设计/告警收敛机制` | 命中 | ~0.88 |
| 全换近义词 | `报警系统/告警降噪` | `告警系统设计/告警收敛机制` | 命中 | ~0.85 |
| 4层每层都偏 | `核心架构/告警后端/告警引擎/熔断` | `核心模块架构/告警后端模块/告警核心引擎/熔断机制` | 命中 | ~0.89 |
| 层级颠倒 | `告警后端/核心架构/熔断机制` | `核心模块架构/告警后端模块/告警核心引擎/熔断机制` | 命中 | ~0.88 |
| 近义词重写 | `监控运维/K8s容器管理` | `部署与运维/Kubernetes集群管理` | 命中 | ~0.89 |
| 3层通知路径 | `告警/通知渠道/钉钉消息推送` | `告警系统设计/通知渠道管理/钉钉集成` | 命中 | ~0.87 |
| 无关查询 | `天气预报` | 无 | 返回但 score 低 | ~0.68 |

### 关键结论
1. **多层级路径偏差完全没问题** — 即使每一层都有偏差，向量语义匹配依然 Rank 1 命中
2. **层级颠倒也能命中** — 向量模型把整段文本当语义理解
3. **噪音过滤有清晰阈值** — 相关匹配 score ~0.85-0.90，无关查询 score ~0.68，**0.75 是安全的分界线**

- 不引入副作用 — 精确匹配仍然优先，向量只是兜底层

## 需求清单

### 标签策略

| 标签 | 存储内容格式 | 用途 |
|------|------------|------|
| `ki-path` | `Group路径: {groupPath} \| 关键词: {kw}` | Group 路径模糊匹配 |
| `ki-relation` | `Relation: {text} \| Group: {groupPath} \| 关键词: {kw}` | Relation 名称模糊匹配 |

### 需求明细

| 优先级 | 需求 ID | 需求描述 | 预期效果 | 依赖 | 验收标准 |
|--------|---------|---------|---------|------|---------|
| P0 | REQ-01 | **路径向量索引写入**：`scan-kb import` 导入时，每条 entry 存两条向量：① `tag=ki-path`（groupPath + keywords）② `tag=ki-relation`（relation text + groupPath + keywords） | Group 和 Relation 均可被向量检索 | - | 导入后两个 tag 均能召回 |
| P0 | REQ-02 | **Group 路径模糊兜底**：`resolveGroupPath` 精确匹配失败后，`mem search <input> --tags ki-path`，score ≥ 0.75 返回近似匹配 + `💡 近似匹配` 提示 | group 路径写错也能命中 | REQ-01 | 偏差路径 Rank 1 命中正确路径 |
| P0 | REQ-03 | **Relation 名称模糊兜底**：`get-module-info` 中 relation 精确匹配失败后，`mem search <input> --tags ki-relation`，score ≥ 0.75 返回近似匹配 + 提示 | relation 名称写错也能命中 | REQ-01 | 偏差 relation 名 Rank 1 命中正确原文 |
| P1 | REQ-04 | **sync-relation 向量同步**：`sync-relation` 写入时同步写入 `ki-path`（如果 group 是新建的）和 `ki-relation` 向量 | 手动写入的路径/relation 也能被模糊搜索 | REQ-01 | sync 后新条目可被召回 |
| P1 | REQ-05 | **增量更新向量同步**：`scan-kb import --mode incremental` 的 add/modify/delete 同步更新 `ki-path` 和 `ki-relation` 向量 | 增量更新后两个索引保持一致 | REQ-01 | add 新增、delete 删除、modify 更新 |
| P2 | REQ-06 | **向量兜底降级策略**：`mem search` 超时或失败时，静默降级为当前的"列出候选项"行为 | 向量服务不可用时不阻塞 | REQ-02 | 超时后返回原有提示 |

## 非功能性约束

- 向量兜底是**可选增强层**，精确匹配仍然优先
- 近似匹配结果必须明确标注 `💡 近似匹配`，不能让用户误以为是精确命中
- `ki-path` / `ki-relation` tag 与现有 `kb-import` category 物理隔离，互不干扰

---

## 潜在风险与注意事项

1. **API 超时**：SiliconFlow embedding API 偶发超时，需有降级策略（REQ-06）
2. **存量数据迁移**：已导入的 KB 数据没有路径向量，需要提供一个补录命令或脚本
3. **路径删除清理**：增量 `delete` 时需要同时删除对应的向量记录
