# 负向需求设计报告

> 分析范围：`src` 目录三个方向 —— A（CLI 参数与误用层）、C（备份恢复与迁移）、D（MCP Server 接口层）。
> 方法论：从现有正向功能反推负向场景，按 7 类分类，P0-P3 风险分级，逐场景给出预防/检测/恢复/引导/容错策略。

## 1. 输入需求（正向需求 → 反推来源）

本次未走 requirement-mining，正向需求来自 `src` 现有功能推导：

| 需求 ID | 范围 | 正向需求描述 |
|---------|------|-------------|
| REQ-A | CLI 参数层 | 用户通过命令行传参调用各命令（store/search/sync-relation/manage-index/setup 等），系统解析并校验参数后执行 |
| REQ-C1 | 备份 | 用户执行 `backup` 对 scope 做 tar.gz 快照 + ai-results 副本 |
| REQ-C2 | 恢复 | 用户执行 `restore --from-snapshot` / `--from-results` 还原数据 |
| REQ-C3 | 迁移 | 用户执行 `migrate-keywords` 批量迁移 keywords 结构 |
| REQ-C4 | 导出 | 用户执行 `export` 导出 KB 内容 |
| REQ-D | MCP 接口 | Agent 通过长驻 MCP Server 调用 11 个工具读写知识库 |

## 4. 详细场景分析（P0/P1 优先）

### NEG-06：多进程/多 MCP 并发写 JSON 覆盖 【P0】

**场景描述**：`relations-cache.json`、`group-index.json` 等 JSON 文件仅靠 WAL 原子写（`.tmp→rename`）保证单文件不损坏，但**无文件锁**。当多个 CLI 进程或多个 MCP 工具调用并发写同一文件时，形成 Last-Write-Wins，先写入的变更被静默覆盖。`sync-relation.ts` 的 `vectorWriteBack`（L449-455）在两次 `writeJson` 间还存在竞态窗口。

**触发条件**：Agent 并发调用多个 `ki_*` 写工具；或用户在两个终端同时跑写命令。

**风险评估**：发生概率中（Agent 场景下并发常见）/ 影响严重（数据丢失且无感知）/ 可检测性低 → **P0**

**程序行为设计**：
- **预防**：写 JSON 前获取文件锁（如向量库已有的 lock 机制推广到 JSON 层）；或采用「读取版本号→CAS 写入」的乐观并发。
- **检测**：写入前比对文件 mtime/内容哈希与读取时是否一致，不一致则重读合并。
- **恢复/引导**：检测到冲突时重试（读-改-写循环，带上限），失败则返回明确错误让调用方重试，而非静默覆盖。
- **UX**：MCP 返回 `{isError:true}` 附「检测到并发写冲突，请重试」；CLI 输出 JSON 错误码 `CONCURRENT_WRITE`。

### NEG-07：磁盘满 / 无写权限无预检 【P0】

**场景描述**：backup/restore/export/store 均无磁盘空间与目录写权限预检，直接依赖系统抛 `ENOSPC`/`EACCES`，用户看到的是原始堆栈或半截产物。

**风险评估**：概率中 / 影响严重（备份/恢复中断可能留下损坏产物）/ 可检测性高 → **P0**

**程序行为设计**：
- **预防/检测**：写操作前预检目标目录可写性（试写探针）与可用空间（估算 tar 体积 vs 可用空间）。
- **恢复**：预检失败立即中止且不产生任何副产物；已产生的 `.tmp` 需清理。
- **UX**：友好错误「磁盘空间不足，需约 X MB，当前可用 Y MB」/「目标目录无写权限：{path}」，退出码非 0。

### NEG-11：restore --from-snapshot 破坏性覆盖 【P0】

**场景描述**：`restoreFromSnapshot`（L72-183）会删除现有 scope 目录再解压。虽已有安全网快照 + 失败自动恢复 + `--yes`/交互确认，但用户误选 scope 或误确认仍会覆盖当前数据。

**风险评估**：概率低 / 影响严重 / 可检测性高 → **P0**（因影响严重）

**程序行为设计**：
- **预防**：确认提示中显式打印「将覆盖 scope=X（含 N 个节点，最后修改于 T）」，让用户核对目标；非交互环境强制要求 `--yes`。
- **检测/恢复**：保留现有安全网快照机制，并在成功后提示安全网快照位置及保留期。
- **UX**：高危红字提示 + 目标摘要 + 明确 y/N 默认 N。

### NEG-01：未知/拼写错参数被静默忽略 【P1】

**场景描述**：commander 未启用 `allowUnknownOption(false)`，如 `--socpe`（拼错）被忽略，命令按默认值执行，用户以为生效。

**风险评估**：概率高 / 影响中 / 可检测性高 → **P1**

**程序行为设计**：
- **预防**：全局启用未知选项报错。
- **引导/容错**：报错时给出最接近的正确参数建议（编辑距离）「未知参数 --socpe，您是否想输入 --scope？」
- **UX**：统一 JSON 错误 `UNKNOWN_OPTION` + 建议项。

### NEG-08：backup 同秒执行覆盖 【P1】

**场景描述**：`makeTimestamp` 精度到秒，同一秒内两次 backup 生成同名 `snapshot.{yyyyMMdd-HHmmss}.tar.gz`，后者覆盖前者。

**风险评估**：概率中（脚本连跑）/ 影响中 / 可检测性高 → **P1**

**程序行为设计**：
- **预防**：时间戳加毫秒或递增序号；或写前检测同名文件存在则追加 `-N`。
- **UX**：输出实际写入的文件名。

### NEG-09：长任务中途放弃（Ctrl-C）【P1】

**场景描述**：backup 打 tar、import 批量写入过程中用户中断，可能残留半截 tar / `.tmp` / 部分写入的 JSON。

**风险评估**：概率中 / 影响中 / 可检测性中 → **P1**

**程序行为设计**：
- **恢复**：捕获 SIGINT，清理未完成的临时产物；已用 WAL 的走 `.tmp→rename` 保证原子性。
- **UX**：「操作已取消，已回滚未完成的写入」。

### NEG-10：向量库被占用降级 【P1】

**场景描述**：向量库被其他进程锁定抛 `CollectionLockedException`，已有优雅降级（仅写 JSON 不写向量）。

**风险评估**：概率中 / 影响中 / 可检测性高 → **P1**（现状良好，补强提示）

**程序行为设计**：
- **恢复**：保持降级，记录「向量写入已跳过，待锁释放后可 `ki manage-index` 重建向量」。
- **UX**：明确告知降级发生及补偿手段。

### NEG-12：migrate-keywords 中途失败无回滚 【P1】

**场景描述**：`migrateScope` 幂等但无事务，逐 scope 写盘，中途失败已迁移部分保留、未迁移部分残留旧结构。

**风险评估**：概率低 / 影响中 / 可检测性中 → **P1**

**程序行为设计**：
- **预防**：迁移前自动 backup（复用 autoBackup）。
- **恢复**：因幂等可安全重跑；失败提示「可重新执行 migrate-keywords 继续」。

## 5. 其余场景（P2/P3）简述

- **NEG-02**：非法 `--depth/--hot-count` 静默回退默认值 + console.warn（query-group.ts L524-552）。建议在回退时显式提示「参数 X 非法，已回退默认值 Y」。
- **NEG-03**：scope 非法字符已被 `validateScope`（scope.ts L18-27，正则 `/^[a-zA-Z0-9_-]+$/`）拦截，路径遍历字符被拒。建议统一错误文案与错误码。
- **NEG-04**：错误输出混用 JSON 与 `console.error`（如 setup.ts L270-277）。建议统一为 `{ok:false,code,message,hint}` 契约。
- **NEG-05**：keywords 超 `maxKeywordCount=50`（constants.ts L42）被 FIFO 静默截断。建议截断时回显告警。
- **NEG-13**：MCP 长驻进程升级源码后仍跑旧代码。建议启动/调用时做版本自检并提示重启。
- **NEG-14**：MCP 工具层无显式超时，同步文件 IO 阻塞事件循环。建议大文件读写异步化并加超时。
- **NEG-15**：manage-index 的 delete 工具不经 MCP 暴露（lib/mcp-tools/manage-index.ts L54），Agent 无法删节点。建议文档说明或受控暴露。

## 7. 设计建议

1. **优先级建议**：先处理 3 个 P0 —— 并发写锁（NEG-06）、IO 前置预检（NEG-07）、恢复二次确认强化（NEG-11），它们涉及静默数据丢失，风险最高。
2. **交互建议**：统一 CLI 错误输出契约（JSON `{ok:false,code,message,hint}`），消除 JSON/console.error 混用（NEG-04）；危险操作一律「目标摘要 + 默认拒绝」确认。
3. **测试建议**：重点补并发写竞态测试（多进程/多 MCP 并发写同一 JSON）、磁盘满/只读目录的故障注入测试、Ctrl-C 中断残留清理测试。
